# Talent Atlas 整体评估

> 项目定位：智能人才搜索引擎，支持自然语言混合检索（结构化 + 非结构化），覆盖项目组队、晋升评估、人才盘点、内部转岗等场景。
>
> **当前状态**：技术方向已确定（方案 A），待细化实施计划。

---

## 一、需求澄清

| 维度 | 内容 |
|------|------|
| **数据源** | Workday + HR Plus → Hive，已有员工信息宽表、标签宽表、考核/peer评价/晋升/目标/主管评价等文本字段 |
| **数据规模** | 3万名员工，历史文本积累深厚，长司龄员工数据量大 |
| **搜索入口** | 自然语言查询，同时包含结构化字段（级别、事业部）和非结构化文本（项目经历、优劣势） |
| **权限模型** | 全员可搜，展示时按行（人员可见性）和列（字段隐私）权限过滤 |
| **用户场景** | 项目组队、晋升评估、人才盘点、内部转岗 |
| **前端现状** | 已有 Talent Card 展示员工基础信息和历史评价，长司龄员工数据展示吃力 |
| **效果要求** | 效果优先（召回质量要有保障），效率通过产品体验改善（可接受 2-3 秒） |

---

## 二、核心难点分析

| 难点 | 说明 | 优先级 |
|------|------|--------|
| **混合检索** | 结构化字段（级别、事业部）和非结构化文本（项目经历、优劣势）如何联合检索、结果如何融合排序 | 🔴 最核心 |
| **长文本 embedding** | 3万员工历史积累丰富，司龄长的员工文本可能非常长（十几年考核、晋升、peer评价），直接 embedding 会超限或失真 | 🔴 关键 |
| **数据质量** | 宽表"不是特别干净"——同一字段在不同源头口径不一致，空值、脏数据如何处理 | 🟡 持续成本 |
| **Query 解析** | 自然语言如何解析为"级别=P6" + "做过新能源项目"这样的混合查询条件 | 🟡 需要 NLP 能力 |
| **权限过滤** | 展示时行列权限过滤，与搜索结果 ranking 的交互——过滤后 ranking 要不要重排 | 🟡 需产品定义 |
| **排序逻辑** | 多场景（组队/晋升/盘点/转岗）对"合适"的定义不同，是否需要场景化排序 | 🟡 中期演进 |

## 三、技术方案对比

### 方案 A：Elasticsearch + 向量数据库 双轨混合检索 ✅ 已选定

```
用户 Query
    ↓
┌─────────────────────────────────────┐
│  Query 解析层                        │
│  → 结构化过滤条件（ES filter）        │
│  → 文本语义 embedding（向量库 ANN）    │
└─────────────────────────────────────┘
    ↓                    ↓
  ES BM25           向量 ANN
 (精准匹配)         (语义相似)
    ↓                    ↓
┌─────────────────────────────────────┐
│  结果融合层（RRF / Rerank）           │
└─────────────────────────────────────┘
    ↓
  Top-K 結果
    ↓
  权限过滤（行+列）
    ↓
  展示
```

- **代表技术**：Elasticsearch + Milvus / Qdrant / Pinecone
- **优点**：各司其职；结构化字段用 ES filter 精确控，向量负责语义；效果最有保障
- **缺点**：两套系统，运维复杂度高；结果融合增加延迟

---

### 方案 B：单一向量数据库 + metadata 过滤（轻量化）

- 所有字段（结构化 + 非结构化）统一做 embedding
- 结构化字段作为 metadata filter 在向量检索时一并过滤
- **代表技术**：Milvus / Qdrant / Weaviate（Weaviate 原生支持 BM25 + vector 混合）
- **优点**：架构简单，一套系统；运维成本低
- **缺点**：metadata filter 在复杂权限场景下不够灵活；长文本切片策略直接影响效果

---

### 方案 C：ES 8.x 原生 Neural Search（一体化）

- ES 8.x 内置向量检索能力，单一索引同时支持 BM25 + kNN
- **代表技术**：Elasticsearch 8.x + native vector fields
- **优点**：复用现有 ES 生态（如果已在用 ES）；运维成本低
- **缺点**：ES 向量能力相比专业向量库有差距；长文本切片能力弱

---

## 四、方案决策

### ✅ 已选定：方案 A（ES + 向量库双轨混合检索）

**选定理由：**
1. 数据在 Hive，引入向量库是新增组件，不影响现有体系
2. 结构化字段精确过滤交给 ES 是成熟方案，团队已有 ES 经验
3. 语义召回交给专业向量库（Milvus / Qdrant），效果最有保障
4. 双轨融合（RRF rerank）可以实现精细的排序控制
5. 团队有 RAG 调用经验，对 embedding 概念不陌生，向量库学习成本可控

**中期演进**：效果稳定、团队熟悉后，可考虑方案 C 合并到 ES，减少运维负担。

## 八、后续步骤建议

1. **数据探查**：对 Hive 中的文本字段做一次数据探查，了解文本长度分布、空值率、语言类型
2. **向量库选型评测**：Milvus Cloud / Qdrant Cloud / Pinecone 对比，选一个最适合的
3. **Embedding 模型评测**：用小样本评测 m3e / text2vec / bge 在人才文本上的效果
4. **Demo 验证**：用小样本数据（100-500 条）跑一个端到端 Demo，验证混合检索效果
5. **Query 解析链路设计**：规则 + 小模型 或 纯 LLM 路由
6. **数据清洗策略**：确定哪些字段需要清洗、清洗粒度
